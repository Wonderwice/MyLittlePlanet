How the Clouds Work

Raymarching a spherical shell, Beer-Lambert extinction, and the Henyey-Greenstein phase function

1. Why Volumetric Clouds?

The planet uses vertex displacement to shape its terrain: every point on the mesh is pushed outward by a noise value, creating mountains and valleys as actual geometry. Clouds cannot work this way. A cloud is not a surface — it is a diffuse mass of ice crystals and water droplets that light can partially penetrate, scatter through, and exit on a different path. Its opacity varies continuously from wispy edges to dense core, and it has no sharp boundary.

Two common shortcuts exist in real-time rendering. The first is a billboard: a flat, textured quad that always faces the camera. Billboards work for isolated distant clouds but collapse into flat cards when viewed from the side. The second is a shell texture: a spherical mesh with a cloud image painted on it. This looks like a rigid shell rather than a floating volume, and it cannot cast self-shadows or interact with light in a physically meaningful way.

Volumetric raymarching avoids both shortcuts. The fragment shader traces a ray from the camera through the cloud volume, sampling density at equally-spaced intervals. At each sample it accumulates how much light is scattered toward the camera and how much is absorbed. The result is a physically-motivated image where clouds thin at their edges, cast shadows on themselves, and glow around the sun.

2. The Spherical Shell

The cloud layer is a spherical shell — the region between two concentric spheres centred on the planet. The inner radius sits just above the highest terrain peak (planet radius + maximum noise displacement + a small gap to prevent interpenetration). The outer radius is the inner radius plus the cloud height:

rinner = rplanet + Anoise + 0.02    router = rinner + hcloud

Both radiuses are recomputed every frame. When the user raises the planet radius or the noise amplitude, the cloud shell rises automatically.

Ray-sphere intersection

Given a ray ro + t·rd (origin + normalized direction), a point lies on a sphere of radius r when:

|ro + t·rd|² = r²

Expanding (using |rd|² = 1 since rd is normalized) gives a quadratic in t:

t² + 2b·t + c = 0    where    b = ro·rd,   c = |ro|² - r²

The half-discriminant form avoids an explicit factor of 2 in the denominator. Two solutions exist when b² - c > 0; the smaller root is the entry point, the larger is the exit point: 

vec2 raySphereIntersect(vec3 ro, vec3 rd, float r) {
  float b    = dot(ro, rd);
  float c    = dot(ro, ro) - r * r;
  float disc = b*b - c;
  if (disc < 0.0) return vec2(-1.0);   // miss
  float sq = sqrt(disc);
  return vec2(-b - sq, -b + sq);
}

Clipping the march to the shell

With two sphere tests we derive the exact interval of t values that lies within the cloud shell. The camera is always outside both spheres, so the outer sphere gives an entry and exit, while the inner sphere (if hit) gives the point where the ray would enter the planet — which is where cloud marching must stop: 

bool raySphereShellIntersect(vec3 ro, vec3 rd,
                              float rInner, float rOuter,
                              out float tEnter, out float tExit) {
  vec2 outer = raySphereIntersect(ro, rd, rOuter);
  if (outer.y < 0.0) return false;      // misses cloud layer entirely

  vec2 inner = raySphereIntersect(ro, rd, rInner);
  tEnter = max(outer.x, 0.0);            // clamp to camera position
  tExit  = (inner.x > 0.0) ? inner.x : outer.y;
  return tExit > tEnter;
}

When inner.x > 0 the ray is heading toward the planet. Setting tExit = inner.x means the march stops exactly at the planet surface — no depth buffer read required. Planet occlusion is handled purely in math.

Geometry, not depth buffer: WebGL 1.0 cannot read from the depth buffer without the WEBGL_depth_texture extension, which is not universally available. The inner-sphere intersection gives us the same answer — the point where the cloud ray would hit the opaque planet — without any texture access.

3. The Density Field

Inside the shell, every point in 3D space has a density — a non-negative scalar that represents how thick the cloud is at that location. Density zero is clear air; density one is the thickest possible cloud. The density field is entirely procedural: no textures are involved.

3D FBM in world space

The density is derived from Fractional Brownian Motion (FBM) evaluated at the world-space 3D position of the sample point. Using world space is what makes the clouds volumetric — two rays that cross the same physical location in the shell see the same density, even if they arrive from different camera angles.

The FBM sums 6 octaves of value noise at baked-in constants (lacunarity = 2, gain = 0.5): 

float fbm(vec3 p) {
  float val=0., amp=0.5, freq=1., norm=0.;
  for (int i = 0; i < 6; i++) {
    val  += amp * valueNoise(p * freq);
    norm += amp;
    freq *= 2.0;
    amp  *= 0.5;
  }
  return val / norm;   // normalized to [-1, 1]
}

This is the same value noise and FBM used in the planet's vertex shader. For cloud self-shadowing, a cheaper 3-octave variant (fbmFast) is used — the coarser approximation is sufficient for shadow estimation.

Coverage threshold

Raw FBM noise is symmetric around zero: half the domain is positive, half negative. The u_coverage slider shifts a threshold that determines how much of that distribution counts as cloud. With threshold τ = 1 - coverage:

density(p) = max(0,   FBM(p) - τ) · udensity · falloff(h)

At coverage 0.45, the threshold is 0.55: only the top 22% of the FBM distribution produces cloud, yielding scattered cumulus. At coverage 0.8, the threshold is 0.2: 60% of the domain is cloud, producing a dense overcast layer.

Height falloff

A uniform shell of identical density would look like a thick, flat fog band. Real clouds are denser in the middle of the layer and fade toward both boundaries. The normalized height within the shell is:

h = (|p| - rinner) / (router - rinner)    ∈ [0, 1]

The falloff multiplier uses two opposing smoothstep curves — one fading in from the inner edge and one fading out at the outer edge: 

float falloff = smoothstep(0.0, 0.15, h)   // ramp up from inner edge
              * smoothstep(1.0, 0.85, h);  // ramp down at outer edge

The 0.15 and 0.85 thresholds mean the top and bottom 15% of the shell fade to zero. The flattest, most opaque clouds sit in the middle third of the shell.

Cloud animation

The sample position is shifted by a time-varying scroll vector before FBM is evaluated, creating steady drift: 

vec3 scroll = vec3(u_time * u_cloudSpeed,  0.0,
                    u_time * u_cloudSpeed * 0.7);
float noise = fbm((p + scroll) * u_cloudFreq);

Using different scroll rates in X and Z (1.0 and 0.7) breaks the symmetry — clouds drift diagonally, more naturally than a purely horizontal scroll. The Y axis is left fixed (no vertical drift, as in real tropospheric clouds).

4. Raymarching — Walking Through the Volume

Between the entry point tenter and exit point texit, the shader takes 64 equally-spaced steps along the ray, sampling the density at each one. The step size is:

Δt = (texit - tenter) / 64

With a default cloud height of 0.12 world units, Δt ≈ 0.002. Each sample is at position p = ro + t · rd.

Jittered start — eliminating banding

Fixed equal steps produce visible concentric rings ("banding") where empty steps and dense steps fall on a regular grid that coincidentally aligns with the shell geometry. The fix is to offset the starting position by a random fraction of one step: 

float jitter = hash3(vec3(gl_FragCoord.xy, u_time)) * stepSz;

hash3 is the Dave Hoskins hash function that also appears in the planet's vertex shader. Using gl_FragCoord.xy (pixel coordinates, unique per fragment) and u_time (changing each frame) produces a different offset for every pixel every frame. Temporally the pattern appears as high-frequency noise, which is perceptually far less distracting than coherent banding rings.

Early termination

Two conditions allow the loop to stop before completing all 64 steps: 

if (t >= tExit || T < 0.01) break;
// ...
if (length(p) < u_cloudInner) break; // planet surface

The transmittance threshold T < 0.01 exits when the accumulated cloud is 99% opaque — further samples cannot contribute more than 1% to the final colour. The inner-sphere check catches any floating-point imprecision that places a step slightly inside the planet despite the shell clipping computed earlier.

5. Beer-Lambert Extinction

As light travels through a participating medium, it is absorbed and scattered out of the beam. The Beer-Lambert law describes the fraction of light that survives after travelling distance d through a medium with extinction coefficient alpha:

T = e-alpha · d

T is the transmittance: the probability that a photon survives the path. At T = 1 the medium is completely transparent; at T = 0 it is completely opaque.

For a heterogeneous volume (density varying at each point), the law generalises to an integral of the extinction coefficient along the path. The raymarcher approximates this with a discrete sum, multiplying transmittance by a factor per step: 

float absorb = density * stepSz * 40.0;   // optical depth for this step
T *= exp(-absorb);

The constant 40.0 is the extinction scale — it sets how quickly cloud density translates into opacity. At density = 1 over a shell of thickness 0.12, the total optical depth is 1 x 0.12 x 40 = 4.8, giving transmittance e-4.8 ≈ 0.008 — nearly fully opaque. The scale was chosen so that moderately dense clouds (density ≈ 0.3) over a default shell produce partial transparency, giving the characteristic layered look of cumulus from space.

Radiance accumulation

Light scattered toward the camera at each step contributes to the final pixel colour in proportion to the transmittance that was present before the step absorbed it: 

L += T * cloudCol * absorb;

Reading T before updating it is physically correct: the light that scatters toward the camera at this depth must first survive all the cloud in front of it (already captured in the current T), not the additional cloud at this step.

The final gl_FragColor is vec4(L, 1.0 - T). The alpha channel is one minus the final transmittance — a fully opaque cloud (T → 0) gives alpha 1.0; a wispy cloud (T → 1) gives alpha 0.0. This value is composited over the planet frame using standard alpha blending with depthMask(false) so cloud fragments do not overwrite the planet's depth values.

6. Lightmarching — Self-Shadow

A cloud that ignores the density between itself and the sun looks uniformly bright — there is no distinction between the sunlit top surface and the shadowed underside. Lightmarching corrects this by tracing a secondary ray from each sample point toward the sun and accumulating the optical depth along that path. 

float lightmarch(vec3 p) {
  float stepSz = (u_cloudOuter - u_cloudInner) / 6.0;
  float shadow = 0.0;
  for (int i = 0; i < 6; i++) {
    p      += u_lightDir * stepSz;
    shadow += sceneDensityFast(p) * stepSz;
  }
  return exp(-shadow * 40.0);  // Beer-Lambert transmittance toward sun
}

The march takes 6 steps (versus 64 for the primary ray), using the 3-octave fbmFast density approximation instead of the 6-octave one. Accuracy is traded for performance: a coarser shadow estimate is still visually correct because human perception of self-shadowing in clouds is forgiving.

The return value is the fraction of sunlight that survives from the sample point to open sky. A value of 1 means fully lit; values near 0 mean the sample sits deep inside a thick cloud mass with almost no direct sun.

The shadow factor in the radiance model

The primary march multiplies the incoming sun light by the shadow factor before computing the cloud colour: 

float sunLight = lightmarch(p);
float lum      = clamp(sunLight * phase, 0.0, 1.0);
vec3  cloudCol = mix(vec3(0.50, 0.55, 0.65),  // deep shadow
                      vec3(1.00, 0.98, 0.95),  // full sunlight
                      lum);

The dark endpoint is a cool blue-grey (the colour of diffuse skylight in shadow); the bright endpoint is a warm white (direct sunlight). A cloud in full shadow gets the sky colour; a cloud directly illuminated gets warm white. The mix parameter lum combines both the shadow and the angular scattering effect described in the next section.

7. The Henyey-Greenstein Phase Function

When a photon hits a water droplet in a cloud, it scatters. The probability that it scatters into any particular direction is given by the phase function. Cloud droplets — being much larger than visible-light wavelengths — scatter much more light forward (in the direction the photon was originally travelling) than backward. This is why you see a bright halo around the sun when looking through thin clouds: forward-scattered sunlight is aimed almost directly toward the camera.

The Henyey-Greenstein phase function (1941) models this angular distribution with a single asymmetry parameter g ∈ (-1, 1):

p(cos θ) = (1 - g²) / (1 + g² - 2g cos θ)3/2

Here cos θ = rd · lightDir, the dot product between the camera ray direction and the sun direction. When g = 0 the phase is isotropic — equal scattering in all directions. When g > 0 the peak is at cos θ = 1 (ray heading toward the sun), producing forward scattering. At g = 0.65 (the value used here) the peak is about 39x stronger than the trough — a pronounced forward-scattering lobe.

Normalization and clamping

The analytic form is normalized to integrate to 1 over the full sphere, which makes each individual value very small (around 0.02-0.3). In a physically correct renderer these values are appropriate because they represent a probability density per steradian. Here we normalize the function so that side-scatter (cos θ = 0) equals 1.0, giving phase a useful range without depending on the solid angle of the sun: 

float henyeyGreenstein(float cosTheta, float g) {
  float g2    = g * g;
  float p     = (1.0 - g2) / pow(max(0.0001, 1.0 + g2 - 2.0*g*cosTheta), 1.5);
  float pSide = (1.0 - g2) / pow(1.0 + g2, 1.5);   // value at cosθ = 0
  return clamp(p / pSide, 0.0, 3.0);
}

After normalization, the function ranges from about 0.4 (backscatter, looking away from sun) through 1.0 (side, perpendicular to sun) up to a clamped maximum of 3.0 (forward, looking toward sun). The clamp prevents the forward peak from blowing out to 39x — a bright halo effect is preserved, but the maximum brightness stays within the display range.

8. Ray Reconstruction from Screen Coordinates

The cloud shader runs in a fullscreen triangle — a single triangle covering the entire viewport with three vertices at NDC positions (-1, -1), (3, -1), and (-1, 3). The fragment shader receives only the NDC position v_ndc; it must reconstruct which world-space ray corresponds to this pixel without any geometric vertex data.

The forward projection pipeline

The standard rendering pipeline transforms world-space positions into screen pixels through three matrices in sequence:

pclip = P · V · pworld

V is the view matrix (world → camera space) and P is the perspective projection matrix (camera space → clip space). After the perspective divide (dividing by the w component), clip-space coordinates become the NDC (Normalized Device Coordinates) visible in the fragment shader.

To go backward — from a pixel to a world-space ray — we apply the inverse matrices in reverse order. Both inverses are computed on the CPU each frame and uploaded as uniforms, since WebGL 1.0 GLSL has no built-in inverse() function.

Inverting the perspective matrix

The perspective matrix produced by mat4Perspective has a known sparse structure — only five entries are non-zero. Its inverse exploits this sparsity analytically: 

function mat4InversePerspective(m) {
  const out = new Float32Array(16);
  out[0]  =  1.0 / m[0];              // aspect / f
  out[5]  =  1.0 / m[5];              // 1 / f
  out[11] =  1.0 / m[14];             // row3, col2
  out[14] =  1.0 / m[11];             // = -1  (row2, col3)
  out[15] = -m[10] / (m[11] * m[14]); // row3, col3
  return out;
}

Inverting the view matrix

The look-at view matrix is a rigid-body transform: a pure rotation followed by a translation. For any rotation matrix R (orthonormal, det = +1), its inverse equals its transpose. The translation part of the inverse is then -RT · t: 

function mat4InverseRigid(m) {
  const out = new Float32Array(16);
  // Transpose the 3x3 rotation block (column-major swap)
  out[0]=m[0]; out[4]=m[1]; out[8] =m[2];
  out[1]=m[4]; out[5]=m[5]; out[9] =m[6];
  out[2]=m[8]; out[6]=m[9]; out[10]=m[10];
  // New translation = -Rᵀ · original translation
  out[12]=-(m[0]*m[12]+m[1]*m[13]+m[2]*m[14]);
  out[13]=-(m[4]*m[12]+m[5]*m[13]+m[6]*m[14]);
  out[14]=-(m[8]*m[12]+m[9]*m[13]+m[10]*m[14]);
  out[15]=1.0;
  return out;
}

Reconstructing the ray in the fragment shader

With both inverse matrices available as uniforms, the fragment shader reconstructs the world-space ray direction in three steps: 

vec3 reconstructRayDir() {
  // 1. NDC point at the far plane
  vec4 clip    = vec4(v_ndc, 1.0, 1.0);

  // 2. Unproject to view (camera) space via inverse projection
  vec4 view4   = u_invProj * clip;
  vec3 viewDir = view4.xyz / view4.w;  // perspective divide

  // 3. Rotate to world space via inverse view (w=0 → rotation only)
  vec4 world4  = u_invView * vec4(viewDir, 0.0);
  return normalize(world4.xyz);
}

Setting w = 0.0 in step 3 applies only the rotation part of the inverse view matrix. The camera translation (the position of the eye in world space) is not part of the ray direction — it is the ray origin, which is already available as u_eyePos.

Why the far plane? Using z = 1.0 (far plane in NDC) rather than z = -1.0 (near plane) gives a view-space z coordinate equal to -far. Dividing by w (which evaluates to a small constant near 0.01) scales the XY components up proportionally, but after normalization the direction is identical regardless of which depth value is chosen.

9. The Render Pass

The cloud system adds a second WebGL program executed as a post-pass after the planet is drawn. It draws exactly three vertices — a single triangle — which covers the entire viewport. This is the fullscreen triangle trick: instead of a quad (two triangles, six vertices), one oversized triangle is sufficient and avoids the diagonal seam that appears in a quad under certain rasterisation rules. 

// CPU: three NDC corner positions — one triangle covering the whole screen
const triVbo = gl.createBuffer();
gl.bufferData(gl.ARRAY_BUFFER,
  new Float32Array([-1,-1, 3,-1, -1,3]),
  gl.STATIC_DRAW);

GL state for compositing

Clouds must blend over the opaque planet rather than replace it. The render pass changes three GL state flags:

FlagPlanet passCloud passWhy
DEPTH_TESTonon Cloud triangle at NDC z = 0 is always in front; test has no effect but must stay on to preserve the planet's depth values for any subsequent passes.
depthMasktruefalse Cloud fragments must not overwrite the planet's depth values. Without this, sky pixels behind transparent cloud edges would record a depth of near-zero, breaking any future depth-dependent pass.
BLENDoffon Standard porter-duff over: SRC_ALPHA, ONE_MINUS_SRC_ALPHA. The cloud alpha (1 - transmittance) mixes the cloud radiance with the planet colour already in the framebuffer.

After the cloud draw call, all three flags are restored to their planet-pass state so the render loop is clean for the next frame.

10. Performance Decisions

Volumetric rendering is GPU-heavy. In the worst case, every pixel on screen runs 64 primary density samples plus up to 6 x 6 shadow density samples per primary step — about 450 noise evaluations per pixel. Several choices reduce the practical cost.

Two noise budgets: full and fast

The primary density uses 6-octave FBM for accurate cloud structure. Shadow density uses 3-octave fbmFast. The shadow path has 4x fewer hash evaluations per step and the coarser result is invisible to the eye, since humans have weak sensitivity to shadow softness inside clouds. This alone cuts the lightmarch cost roughly in half.

Adaptive march length

For a ray that clips only a thin sliver of the shell (grazing angle), the shell intersection returns a short interval and Δt is tiny. The ray exits long before consuming all 64 steps. For a ray that plunges through the centre of the cloud layer, Δt is larger — each step sees more density — and the transmittance threshold T < 0.01 typically triggers early in a dense cloud, aborting the remaining steps.

Planet hit as an early exit

For pixels where the ray intersects the inner sphere before the outer sphere (e.g., looking at the edge of the planet just above the atmosphere), the shell intersection sets tExit = inner.x, making the march interval short or zero. An additional in-loop check if (length(p) < u_cloudInner) break provides a safety net: if a floating-point step overshoots the inner boundary, the loop exits rather than sampling planet-interior density.

Lightmarch only on dense steps

The 6-step shadow march is only launched when the primary density exceeds 0.001: 

if (density > 0.001) {
  float sunLight = lightmarch(p);
  // ... accumulate
}

In clear sky regions between cloud cells, density is exactly zero. Those steps take only the sphere-intersection check and one density evaluation — no lightmarch, no radiance accumulation. The GPU threads handling those steps finish early, freeing warp slots for threads still inside a cloud.
