How the Planet Works

Parametric surfaces, FBM noise, domain warping, and the honest limits of vertex displacement

Contents 1. UV Sphere 2. Cube-Sphere 3. Vertex Displacement 4. FBM Noise 5. Domain Warping 6. Adaptive LOD 7. Limits vs Ray Marching

1. Parametric Surfaces and the UV Sphere

A parametric surface maps a rectangle in parameter space to a surface in 3D. Instead of specifying points directly, you write a function P(u, v) : [0,1]² → ℝ³ and then sample it on a regular grid. Every point on the surface has a unique (or near-unique) pair of coordinates (u, v), analogous to latitude and longitude on Earth.

A UV sphere (also called a longitude–latitude sphere) is the simplest parametric sphere. Define:

φ = 2π·u   (azimuth angle, 0 → 2π)
θ = π·v   (polar angle, 0 at north pole → π at south pole) P(u,v) = ( sin θ cos φ,   cos θ,   sin θ sin φ )

The Y axis is up. At θ = 0 all φ values map to the same north-pole point (0, 1, 0), and at θ = π they all map to (0, -1, 0). This is the pole singularity — the UV grid degenerates at the poles, creating a "pinched" region. It produces slightly denser triangles near the poles and slightly stretched ones near the equator, but for our purposes it is perfectly usable.

Building the mesh

We divide the unit square [0,1]² into a RES x RES grid. With RES = 128 we need (128+1) x (128+1) = 16,641 vertices and 128 x 128 x 2 = 32,768 triangles. The extra row and column (hence RES+1, not RES) is the fence-post rule: a fence with 128 intervals needs 129 posts.

Each vertex stores only its two UV coordinates. The vertex shader reconstructs the 3D position. The index buffer for one quad at row i, column j is: 

// tl = top-left, tr = top-right, bl = bottom-left, br = bottom-right
tl = i * (RES+1) + j
tr = tl + 1
bl = (i+1) * (RES+1) + j
br = bl + 1

// Two triangles, CCW winding when viewed from outside the sphere:
Triangle 1: tl → tr → bl
Triangle 2: tr → br → bl

CCW (counter-clockwise) winding matters because WebGL uses it to identify front faces. We can verify this analytically: the cross product of the phi-direction tangent ∂P/∂φ = (-sin θ sin φ, 0, sin θ cos φ) with the theta-direction tangent ∂P/∂θ = (cos θ cos φ, -sin θ, cos θ sin φ) points outward from the sphere — exactly what back-face culling needs.

The seam: At u = 0 and u = 1, the φ values are 0 and 2π, which map to the same 3D positions. We create duplicate vertices at the seam — the last column shares world-space position with the first column, but has a different UV coordinate (u = 1 vs u = 0). This duplication is harmless and keeps the UV mapping seamless.

2. Cube-Sphere: No Poles, No Seams

The UV sphere has two structural weaknesses that matter for a terrain renderer. First, the triangles at the poles are tiny and the triangles at the equator are stretched — the mesh is not uniform. Second, the finite-difference normal computation breaks at the poles, where all φ values collapse to the same 3D point and the derivative ∂P/∂φ → 0.

The cube-sphere fixes both problems. Start from a cube: six faces, each a regular grid of quads. Normalize every vertex position (divide by its length), projecting it onto the unit sphere. The result is a sphere whose triangle density varies by at most ~27% across the surface — far more uniform than a UV sphere.

Psphere = Pcube / |Pcube|

Building the six faces

Each face is described by three orthogonal unit vectors: fwd (the outward-facing normal of the cube face), r (right across the face), and u (up across the face). A parameter pair (s, t) ∈ [-1, 1]² sweeps the face: 

Pcube(s, t) = fwd + r·s + u·t

After normalization these six patches tile the sphere without gaps or overlaps. The right and up vectors for each face are chosen so that cross(r, u) = fwd, ensuring counter-clockwise triangle winding when viewed from outside the sphere.

Facefwdr (right)u (up)
+X(1, 0, 0)(0, 0, -1)(0, 1, 0)
-X(-1, 0, 0)(0, 0, 1)(0, 1, 0)
+Y (top)(0, 1, 0)(1, 0, 0)(0, 0, -1)
-Y (bottom)(0, -1, 0)(1, 0, 0)(0, 0, 1)
+Z(0, 0, 1)(1, 0, 0)(0, 1, 0)
-Z(0, 0, -1)(-1, 0, 0)(0, 1, 0)

The a_dir vertex attribute

Instead of UV coordinates, each vertex stores its unnormalized cube direction as a vec3 attribute (a_dir). The vertex shader normalizes it to get the sphere point: 

vec3 n0 = normalize(a_dir);

This is more natural than a UV parameterization because the noise functions already operate in 3D space — no trigonometry needed to convert (u, v) to a 3D sample position.

The Duff 2017 tangent frame

The UV sphere's finite-difference normals were computed by offsetting in UV directions (Δφ, Δθ). The cube-sphere has no UV parameterization, so offsets must instead be taken in the sphere-tangent plane. Given a sphere point n, we need two orthogonal tangent vectors t and b such that cross(t, b) = n (outward normal convention). The classic construction t = cross(n, worldUp) breaks whenever n is nearly parallel to worldUp. The Duff 2017 method is singularity-free everywhere on the sphere: 

void duffFrame(vec3 n, out vec3 t, out vec3 b) {
  float s = (n.z >= 0.0) ? 1.0 : -1.0;
  float a = -1.0 / (s + n.z);
  float c = n.x * n.y * a;
  t = vec3(1.0 + s*n.x*n.x*a,  s*c,  -s*n.x);
  b = vec3(c,  s + n.y*n.y*a,  -n.y);
}

The sign variable s selects a branch per hemisphere. Neither branch can produce a zero denominator: when s = +1 we need n.z ≠ -1 (which is the south pole of the z-axis, impossible since s = +1 only when n.z ≥ 0), and vice versa for s = -1.

With the tangent frame, finite-difference normals become a straightforward world-space computation: 

// Offset sphere points (re-normalize to stay on unit sphere)
vec3 nt = normalize(n0 + EPS * t);
vec3 nb = normalize(n0 + EPS * b);

// Displaced world positions
vec3 d0 = (radius + displace(n0)) * n0;
vec3 dt = (radius + displace(nt)) * nt;
vec3 db = (radius + displace(nb)) * nb;

// Outward surface normal
vec3 N = normalize(cross(dt - d0, db - d0));
Index buffer safety: WebGL 1 index buffers are 16-bit (Uint16Array, max index 65 535). A 128x128 face patch has (128+1)² = 16 641 vertices — safe. The renderer issues six separate draw calls (one per face), so each call's maximum index stays well below 65 535 at every resolution.

3. Vertex Displacement

Once we have a smooth sphere, we want mountains and valleys. The key idea is simple: instead of rendering the sphere at radius r, move each vertex outward (or inward) by a height value h derived from a noise function:

Pdisplaced(u, v) = (r + h(P̂)) · P̂(u, v)

where P̂ is the unit sphere point (radius 1). The displacement h is evaluated at the undisplaced sphere surface, not at the displaced position, so there is no feedback loop.

This gives genuine 3D terrain because the vertices physically move — unlike a normal map or parallax mapping technique which only simulates depth via shading tricks without changing geometry.

Recomputing normals

After displacement the sphere surface has changed shape, so its normals have changed too. If we kept the original sphere normals, mountains would be lit as if they were smooth bumps. We need the normals of the displaced surface.

The cleanest approach is finite differences: sample the displacement at three nearby UV positions and approximate the tangent vectors: 

vec3 p0 = spherePt(u,       v);      // centre sample
vec3 pu = spherePt(u + eps, v);      // offset in φ direction
vec3 pv = spherePt(u,       v + eps);// offset in θ direction

vec3 d0  = (radius + displace(p0)) * p0;
vec3 dpu = (radius + displace(pu)) * pu;
vec3 dpv = (radius + displace(pv)) * pv;

// Surface tangents → outward normal
vec3 N = normalize(cross(dpu - d0, dpv - d0));

The epsilon eps = 0.003 is in UV space, corresponding to about 0.6° of arc on the sphere. Too small and floating-point noise dominates; too large and the derivative approximation is crude. For the noise frequencies used here, 0.003 gives accurate normals.

Pole singularity: At v = 0 or v = 1, all φ values map to the same 3D point, so dpu - d0 → 0 and cross(…) becomes undefined. The fix is simple: clamp v to [2·eps, 1 - 2·eps] when sampling for finite differences. The pole vertices are still rendered at their correct positions; only the derivative samples are clamped.

4. FBM Noise

Plain random noise produces a rough, salt-and-pepper pattern with no large-scale structure. Real terrain has features at every scale simultaneously: continental plains, mountain ranges, hills, boulders. Fractional Brownian Motion (FBM) achieves this by summing multiple octaves of noise at increasing frequencies and decreasing amplitudes.

Value noise

The building block is value noise:

  1. Evaluate a hash function at each corner of the integer lattice cube containing P.
  2. Trilinearly interpolate the eight corner values using a smooth kernel.

We use the quintic smoothstep 6t⁵ - 15t⁴ + 10t³ as the interpolation curve. It has zero first and second derivatives at the lattice boundaries, giving C² continuity — smoother than the classic cubic 3t² - 2t³ and free of the derivative discontinuity that shows up as shading "creases" on curved surfaces.

For the hash function we use the Dave Hoskins approach: 

float hash3(vec3 p) {
  p  = fract(p * 0.3183099 + vec3(0.1));
  p *= 17.0;
  return fract(p.x * p.y * p.z * (p.x + p.y + p.z));
}

This avoids sin(), which has documented precision issues as a hash function on some GPU drivers (the mantissa bits cycle rather than appear random at large inputs).

Layering octaves

FBM accumulates N octaves, each contributing noise at a different scale:

FBM(p) = Σk=0N-1 gaink · noise( p · lacunarityk )

Lacunarity (typically 2.0) multiplies the frequency per octave — each layer sees features twice as fine as the previous. Gain (typically 0.5) multiplies the amplitude per octave — fine details contribute less than broad shapes. Together, the amplitude halves every time the frequency doubles, producing the same statistical self-similarity as a Brownian surface (fractal dimension ≈ 2).

OctaveRelative frequencyRelative amplitude (gain=0.5)Feature size
01x0.500continents
12x0.250mountain ranges
24x0.125large hills
38x0.063small hills
416x0.031rocks
532x0.016surface texture

We normalize the output by dividing by the sum of all amplitudes (the denominator converges to A0 / (1 - gain) = 1 for gain = 0.5, A₀ = 0.5). This keeps FBM in [-1, 1] regardless of the number of octaves.

5. Domain Warping

Plain FBM produces terrain that looks geologically correct but compositionally bland — the features are statistically isotropic and lack the elongated ridges, sinuous valleys, and swirling patterns seen in real planetary geology. Domain warping (introduced by Inigo Quilez) fixes this by distorting the input coordinate before querying the noise.

How it works

Instead of evaluating fbm(p) directly, we evaluate it at a perturbed position p + offset, where the offset is itself computed from FBM. The result is a noise field that has been "folded" over itself: 

// Single-level warp
vec3 q = vec3(
  fbm(p * warpFreq),
  fbm(p * warpFreq + 5.2),
  fbm(p * warpFreq + 1.3)
);
float result = fbm(p + warpStrength * q);

We use two levels of warping: a second offset vector r is computed from the q-warped position, then the main FBM is sampled at p + warpStrength · r. Two-level warping creates more complex folded structures — overhanging ridges, tightly wound spirals, and the kind of organic complexity that single-level warping can't produce.

The scalar offsets (5.2, 1.7, 9.2, …) shift the three vector channels to different regions of noise space, preventing them from being correlated. Without these offsets, all three channels would produce similar values and the warp would be nearly isotropic — no different from no warp at all.

What the parameters control

ParameterWhat it does
warpStrength How far the coordinate is displaced before sampling. 0 = no warp (pure FBM). Large values create extreme folded patterns and can distort the terrain beyond recognition.
warpFreq Frequency of the warp noise itself. Low values = large, sweeping bends. High values = rapid, tight swirls.
warpOctaves Detail in the warp itself. More octaves = the warp has fine structure on top of its large-scale structure.

Why it can't produce real overhangs

The terrain is still defined as a single-valued function on the sphere surface — one height h per point (u, v). Domain warping makes the noise field more complex, but the surface is still a single-layer shell. If the warp is strong enough, two nearby surface points might have very similar or even identical heights, which looks like a cliff — but there is no material behind the cliff face, no overhang, no cave.

6. Adaptive Level of Detail

A 128x128 mesh looks smooth from a distance of 5 units. Zoom in to 1.5 units and the same mesh shows visible faceting — flat polygons where smooth terrain should be. Zoom out to 8 units and most triangles are sub-pixel, wasting GPU time. Level of detail (LOD) solves both ends of this problem by switching between mesh resolutions according to camera distance.

Pre-built resolution levels

This renderer pre-allocates four cube-sphere meshes at startup, one for each LOD level:

LevelResolutionTriangles/faceTotal trianglesActive when distance ≤
0 (finest)128x12832,768196,6082.0

Each level's vertex buffer (dirData) and index buffer are uploaded to GPU memory once during initialization as gl.STATIC_DRAW — 24 VBOs and 24 IBOs in total, a few megabytes of VRAM at most. The data for all levels is immediately available every frame; no CPU upload occurs during rendering.

Distance-based selection

Each frame, the renderer reads the camera distance and picks the finest applicable level: 

function getLODIndex(dist) {
  for (let i = 0; i < LOD_LEVELS.length; i++) {
    if (dist <= LOD_LEVELS[i].maxDist) return i;
  }
  return LOD_LEVELS.length - 1;
}

The HUD displays the active level alongside the FPS counter (FPS: 60 | LOD: 64). Zoom in past distance 3.5 and the counter immediately shows LOD: 128 — the switch is free because both meshes are already on the GPU.

Why not per-face LOD?

A more sophisticated approach assigns a different resolution to each of the six cube faces based on its distance from the camera. The face closest to the camera gets the finest mesh; the face on the far side of the planet gets the coarsest.

This renderer keeps all six faces at the same level for simplicity — when viewing the whole planet, the error of under-tessellating a distant face is not visible. Per-face LOD becomes worthwhile only when the camera flies into the atmosphere, where some faces are very close and others are behind the visual horizon.

LOD seams: When adjacent patches have different resolutions, cracks appear at their shared edges because the fine mesh has vertices that the coarse mesh does not. Because all six faces share the same resolution here, no seam stitching is required.

7. Limitations vs True Ray-Marched Caves

The technique we use — vertex displacement on a sphere mesh — is excellent for real-time rendering and is what most game planet renderers do. But it has a fundamental topological constraint:

Displacement mapping cannot produce overhangs, tunnels, or caves.
The sphere mesh is a topological sphere — it has no holes. Every direction from the centre intersects the surface exactly once. No amount of vertex displacement can change this topology.
